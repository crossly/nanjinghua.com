# Qwen3-TTS 南京话试音生成链路与下一轮验证方案

研究日期：2026-07-24

用途：回答当前城市故事试音是否由 Qwen3-TTS 生成、现有 VoiceDesign 4-bit 链路能做什么与不能证明什么，并为下一轮候选生成和南京本地使用者盲测提供可执行方案。本记录不评价某一种南京话为唯一标准，也不把合成音当作真实语音样本。

## 结论

1. **当前 15 条试音确实来自 Qwen3-TTS 系列。** 仓库生成脚本加载的模型 ID 是 `mlx-community/Qwen3-TTS-12Hz-1.7B-VoiceDesign-4bit`，通过 `mlx_audio` 的 `generate_audio` 生成 WAV；每句只输入汉字、`Chinese` 语言码，以及“南京本地中年女性、南京话口音”等自然语言说明。[R1]
2. **它不是 Qwen 官方文档所示的原始推理基线。** 当前脚本使用第三方命名空间 `mlx-community`、MLX 运行时和名称带 `4bit` 的模型；Qwen 官方发布表列出的是 `Qwen/Qwen3-TTS-12Hz-1.7B-VoiceDesign`，官方模型 API 将权重参数标为 BF16，官方评测也说明使用 `torch.bfloat16`。因此，官方模型成绩不能直接当作当前 4-bit 转换版的质量证明，也不能在没有对照实验时断言“效果差就是 4-bit 导致的”。[R1][Q1][Q6][Q10]
3. **当前问题不是继续堆一句“更像南京话”的提示就能可靠解决的。** 官方 VoiceDesign 接口公开的条件只有目标文字、语言和自然语言 `instruct`；官方将其能力描述为音色、情绪、韵律等声学属性控制。开源模型表把语言列为 `Chinese`，没有列出南京话专用语言码；九个 CustomVoice 预设中明确列出的汉语方言音色只有北京、成都，没有南京。阿里云 Qwen-TTS 服务文档更直接地写明“声音设计音色”和“声音复刻音色”不支持方言。[Q1][Q2][Q3][Q8]
4. **官方现成南京话路径是系统音色，不是 VoiceDesign。** 阿里云官方音色表列有 `Li`，音色名“南京-老李”，语言标为“中文（南京话）”，支持 Qwen3-TTS-Flash / Realtime 相应系列。这是当前最应加入盲测的官方南京话候选，但它是固定男性角色音色，不自动适合所有故事人物。[Q9]
5. **更有证据的下一步是做四路盲测，而不是直接覆盖现有文件。** 同一批目标短句同时生成：A）当前 MLX 4-bit VoiceDesign；B）Qwen 官方 1.7B VoiceDesign `bfloat16`；C）Qwen3-TTS-Flash 的官方南京话系统音色 `Li`；D）Qwen 官方 1.7B Base，以取得授权的南京本地使用者参考录音做实验性 voice clone。先由本地使用者盲评，再决定是否替换。只有积累了干净、逐字转写、同一说话者且可用于训练的材料后，才进入 Base 单说话者微调。[Q1][Q4][Q5][Q8][Q9]

## 当前仓库实际链路

### 生成

`scripts/generate-nanjinghua-trials.py` 的链路是：

1. `load_model("mlx-community/Qwen3-TTS-12Hz-1.7B-VoiceDesign-4bit")`；
2. 对 15 个场景各选一句目标短句；
3. 统一加入“匿名的南京本地中年女性”“使用南京话口音”等音色说明，再追加场景语气说明；
4. 调用 `generate_audio`，参数含 `lang_code="Chinese"`、`max_tokens=180`；
5. 写入 `public/audio/nanjinghua-trials/{slug}.wav`。[R1]

脚本没有输入真人参考音频、参考音频逐字稿、发音词典或音系标注，也没有记录随机种子、完整采样参数、模型文件修订号或输出清单。现有文件实测均为单声道、24 kHz、16-bit PCM WAV；这只说明交付格式，不说明南京话发音准确度。

### 页面连接与状态

`src/content/city-story-dialogues.ts` 将 15 个文件逐一连到目标短句，并将全部对话标成“待南京本地使用者复核”。政策页也明确说明试音是实验性合成音，不证明发音已经准确。[R2][R3]

因此，用户反馈“大部分效果不好”与仓库当前的公开状态并不冲突：这些文件从未经过能够把它们升级为“已复核南京话”的验证流程。

## VoiceDesign 4-bit 的能力边界

### 可以合理使用的能力

- 根据目标文字和自然语言描述生成一个候选音色、情绪、语速和韵律；
- 为同一句话快速采样多个候选，作为探索和盲测材料；
- 在不模仿具体真人时探索大致年龄、音色和场景语气；
- 作为 A/B 对照的一路，衡量第三方 MLX 4-bit 链路与官方 `bfloat16` VoiceDesign 的实际差异。

这些用途与 Qwen 官方对 VoiceDesign 的公开接口和“自然语言控制音色、情绪、韵律”的说明一致。[Q1][Q2][Q3]

### 不能据此声称的能力

- 不能因为提示中写了“南京话口音”，就声称输出采用了已验证的南京话音系；
- 不能把 `Chinese` 语言码解释成南京话专用语言码；
- 不能把官方 `bfloat16` 评测结果转移到当前 `mlx-community` 4-bit 权重和 `mlx_audio` 运行时；
- 不能在没有同句、同参数对照时把错误归因于量化；
- 不能用 VoiceDesign 本身完成真人 voice clone。Qwen 官方把 VoiceDesign 与 Base voice clone 作为不同模型与不同生成接口；Base 才接收 `ref_audio` 和 `ref_text`。[Q1][Q3][Q4]
- 不能把阿里云 `Li` 系统音色对南京话的支持反推到开源 VoiceDesign；它们是不同产品入口和不同音色条件。[Q8][Q9]

“4-bit”目前只是仓库所用模型 ID 的一部分；仓库没有保存该转换的量化方法、校准数据或与官方权重的质量对照。因此本项目能验证的是**输出差异**，不能从名称反推出具体损失机制。[R1]

## 为什么自然语言提示不能保证南京话发音

这里不是说自然语言提示“完全无效”，而是它缺少可核验的发音约束：

1. **开源 VoiceDesign 接口的语言粒度不足。** VoiceDesign 示例传入的是 `language="Chinese"`；官方发布表也只把 Chinese 列为支持语言，没有南京话专用入口。[Q1][Q3]
2. **提示主要描述声学风格。** 官方将指令控制范围表述为音色、情绪、韵律、语速等，并用“撒娇稚嫩的女声”等风格描述举例；这不等于逐音节发音控制。[Q1][Q3]
3. **当前链路没有南京话声音证据。** “南京本地中年女性”“南京话口音”只是文字标签；没有真实说话者录音、逐字稿或发音标注让模型在本次推理时对齐。[R1]
4. **官方预设也没有南京音色。** CustomVoice 列出的九个预设中，北京方言和四川方言各有一个明确标注的说话人，没有南京话预设。换用某个现成 CustomVoice 不能解决南京话发音问题。[Q2]
5. **云服务也明确区分“提示”与“方言音色”。** 阿里云 Qwen-TTS 文档要求方言使用支持方言的系统音色，并将声音设计、声音复刻音色标为不支持方言；系统音色 `Li` 才明确标注南京话。[Q8][Q9]
6. **输出本身具有采样差异。** 官方生成接口公开 `do_sample`、`top_k`、`top_p`、`temperature` 等参数，并说明大多数场景建议采样；当前脚本没有固定或落盘这些条件。一次生成失败不能代表提示的全部可能输出，一次偶然成功也不能代表可重复。[Q7][R1]

所以“精调发音”应被拆成两件事：先用真实南京话材料约束声音，再用盲测决定哪些候选在具体短句和场景中可接受。单纯把提示写得更长，只会增加描述，不会自动增加发音证据。

## 官方提供的三条相关路径

### 1. VoiceDesign：无参考音频的候选探索

官方 1.7B VoiceDesign 使用 `generate_voice_design(text, language, instruct)`；适合做风格候选和建立与当前 4-bit 链路的对照，不应作为南京话准确度的最终依据。[Q3]

### 2. 南京话系统音色 `Li`：下一轮官方基线

阿里云 Qwen-TTS 官方文档要求方言使用支持方言的系统音色，官方音色表将 `Li` 标为“南京-老李”和“中文（南京话）”。因此 `Li` 是下一轮最直接的官方南京话基线。[Q8][Q9]

它的边界同样明确：这是 Qwen3-TTS-Flash / Realtime 服务中的固定男性系统音色，不是开源权重，也不是可由本站任意设定年龄、性别或角色的 VoiceDesign。若发音通过而人物感不合适，可把它保留为发音比较基准，不强行用于所有页面。

### 3. Base voice clone：实验性本地对照

官方 1.7B / 0.6B Base 支持从用户参考音频做约 3 秒快速 voice clone。完整模式同时传 `ref_audio` 与准确的 `ref_text`；如果只用 `x_vector_only_mode=True`，官方明确提示克隆质量可能下降。参考提示可以预先提取并复用于多句生成。[Q1][Q4]

对本项目而言，Base 的实验价值不是“复制某个名人音色”，而是测试一位已授权南京本地使用者的真实语音条件能否改善输出。必须同时写清限制：开源官方文档没有承诺南京话 clone；阿里云 Qwen-TTS 服务文档还明确将声音复刻音色标为“不支持方言”。因此这一路只能由盲测证明具体句子可用，不能预先称为官方南京话方案。[Q4][Q8]

### 4. Base 单说话者微调：有数据后再做

Qwen 官方当前只提供 12Hz 1.7B / 0.6B Base 的**单说话者微调**。训练 JSONL 每条需要目标 WAV、对应文字和参考音频；官方建议全数据使用同一参考音频以提高说话者一致性。多说话者微调仍列为未来能力。官方文档没有给出“多少分钟或多少小时必然足够”的门槛，本项目不得自行把一个任意时长写成官方要求。[Q5]

微调只有在以下条件同时成立后才进入：

- 同一说话者录音数量足以留出未参与训练的测试句；
- 每条录音有人工核对的逐字稿和场景信息；
- 授权明确包含模型训练、内部评测和预期发布方式；
- 原始文件、派生文件与撤回关系可追踪；
- 本地使用者盲测证明 Base clone 仍无法达到目标，而不是短句本身或录音质量有问题。

## 下一轮候选生成方案

以下数字是本项目的实验设计，不是 Qwen 官方门槛。

### 阶段 0：先固定语言目标

1. 从 15 句中选 5 句诊断集，覆盖疑问句、陈述句、句末语气、南京话高频词和容易被普通话化的片段；建议至少包含“阿要辣油啊？”“上哪块去啊？”。
2. 请至少 2 位南京本地使用者先核对目标文字、实际场景和可接受变体。文字未通过时不生成音频。
3. 由 1 位取得明确授权的南京本地使用者录制：
   - 5 句目标句，作为不喂给模型的人类参考；
   - 1 至 3 条不同内容的干净参考句，附逐字稿，作为 Base clone 输入。
4. clone 参考句不要包含本轮目标句，避免把复述目标句误当成对未见文字的发音泛化。

### 阶段 1：建立四路可归因候选

VoiceDesign 和 Base clone 每句各采样 3 条，固定系统音色 `Li` 每句生成 1 条，共 `5 × (3 + 3 + 1 + 3) = 50` 条：

| 路径 | 模型与运行时 | 条件 | 要回答的问题 |
| --- | --- | --- | --- |
| A | 当前 `mlx-community` VoiceDesign 4-bit + `mlx_audio` | 保持现有提示，显式记录全部采样参数与随机种子 | 当前结果能否通过多次采样改善 |
| B | 官方 `Qwen/Qwen3-TTS-12Hz-1.7B-VoiceDesign` + 官方 `qwen-tts`，`bfloat16` | 与 A 使用相同目标文字和等义提示 | A/B 差异是否指向量化或运行时链路 |
| C | 阿里云 Qwen3-TTS-Flash 系统音色 `Li` | 使用官方标注的“中文（南京话）”音色，不追加 VoiceDesign 人设 | 官方南京话系统音色能否提供可靠发音基线 |
| D | 官方 `Qwen/Qwen3-TTS-12Hz-1.7B-Base` voice clone | 同一份已授权南京本地使用者参考音频与准确逐字稿 | 官方未承诺方言支持的本地 clone，实际能否改善南京话感知 |

每条候选都写入机器可读清单：候选 ID、目标句、模型精确 ID / revision、运行时版本、参考音频 ID、提示、语言、随机种子、采样参数、输出 SHA-256。不要先覆盖 `public/audio/nanjinghua-trials/`。

如果硬件无法运行官方 `bfloat16` VoiceDesign，B 路标为“未完成”，不能用 A 对 C / D 的结果替代量化对照。若未配置阿里云 Qwen-TTS 服务，C 路也标为“未完成”，不能把任意男性音色冒充 `Li`。

### 阶段 2：南京本地使用者盲测

1. 至少邀请 5 位真实南京本地使用者，记录年龄段、主要成长片区、长期居住经历和家庭语言背景；这些信息用于解释差异，不用于认证“正宗南京人”。
2. 候选改为随机短 ID，隐藏模型、提示和生成路径。同一受试者内随机播放顺序。
3. 所有候选只做一致的响度和首尾静音处理，不做逐句变调或人工修音；否则无法判断模型路径。
4. 每条先让受试者写下“听到的句子”，再显示目标文字并评分：
   - 发音是否像自己熟悉的南京话：1–5；
   - 是否自然，不像普通话硬套词：1–5；
   - 语气是否符合场景：1–5；
   - 哪个字、词或尾音最不对；可接受的另一种说法是什么。
5. 每句允许选择“都不合格”，避免被迫从坏候选中选一个。
6. 人类参考音频也以盲 ID 混入，用来判断评分表和受试者是否能区分真实基准；它不是要求模型必须复制某一个人的全部变体。

### 阶段 3：通过条件与归因

单句候选进入替换池的建议条件：

- 发音南京话感知分中位数至少 4 / 5；
- 至少 4 / 5 受试者认为可用于该场景；
- 没有 2 位或以上受试者指出同一个改变词义或明显普通话化的关键发音问题；
- 结果仍标明复核覆盖的人群和日期，不升级成“唯一正确南京话”。

路径判断：

- **B 明显优于 A：** 优先迁移官方权重 / 推理链路，再讨论是否保留 4-bit；
- **C 的发音明显最好：** 优先采用官方南京话系统音色；若固定男性角色不符合具体场景，可只将 C 作为发音基线；
- **A 与 B 都差、D 明显更好：** 可以继续验证 Base clone，但仍按实验性路线逐句复核，不将结果概括成“Base 支持南京话”；
- **四路都差：** 先修目标文字、参考录音或场景，不进入发布；
- **D 仍不稳定但真实参考清晰：** 才准备单说话者微调小试验，并用完全未参与训练的目标句盲测；
- **评审分歧按年龄或片区成组：** 保留变体说明或多版本，不强行选一个“标准音”。

## 下一步建议

当前 15 条文件先继续保留“AI 合成试音 / 待南京本地使用者复核”标记，不应仅靠新提示批量覆盖。最小可执行下一步是：

1. 先完成 5 句文字复核和一份合规真人参考录音；
2. 写可重复的四路候选生成脚本与清单；
3. 生成 50 条内部候选；
4. 完成盲测并保存匿名化评分；
5. 只替换通过条件的单句，其余继续不发布或保留待复核状态。

这条路径能把“听起来不好”转化为可定位的问题：是文字、VoiceDesign 提示、4-bit / MLX 链路、缺少真人语音条件，还是南京话本身存在不同使用变体。
